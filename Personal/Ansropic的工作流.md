**整套流程的底层原则**（来自 Claude Code 创作者 Boris Cherny 本人）：

Boris 从不在同一个分支上并行工作——每个本地 session 用独立的 git checkout，同时跑 5 个本地加 5-10 个云端 session。他的 CLAUDE.md 只有约 2500 token（100 行），但团队每周更新多次，每次 PR 后都把 Claude 犯的错追加进去。

**最容易被忽略的一条铁律：**当 Claude 做错了某件事，就把它加进 CLAUDE.md，让它不再重犯。这个文件会被 git 追踪，全团队共享。

**为什么要 Plan Mode 先行：**在任何非平凡功能上，先写结构化 spec（范围+验收标准+不做什么），在 Plan Mode 里审查实现计划，由你自己做架构决策——Claude 提议，你批准，然后再执行。这能防止 Claude 花 20 分钟自信地解决一个完全错误的问题。

**为什么 Hooks 比 CLAUDE.md 更可靠：**CLAUDE.md 是建议性的，Claude 大约 80% 的时候会遵守。Hooks 是确定性的，100% 执行。如果某件事必须每次都发生（格式化、lint、安全检查），就做成 hook，不要指望 prompt。
