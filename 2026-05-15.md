## Lets write a hooks 
### 01. Structure 
```{
  "hooks": {
    "<EventName>": [          // Layer 1: which event
      {
        "matcher": "<filter>", // Layer 2: which occurrences of the event
        "hooks": [
          {                    // Layer 3: what to run
            "type": "command",
            "command": "<your-script>"
          }
        ]
      }
    ]
  }
}
```
### 02.Sessions config 

There are ~30 events total. For solo use, focus on these 7:

| Event              | When                        | Common use                                         |
| ------------------ | --------------------------- | -------------------------------------------------- |
| `SessionStart`     | New/resumed session         | Inject git status, current branch, etc.            |
| `UserPromptSubmit` | You hit Enter on a prompt   | Add context based on the prompt; block bad prompts |
| `PreToolUse`       | Before any tool call        | **The big one** — block/modify/allow tool calls    |
| `PostToolUse`      | After a tool succeeds       | Auto-format, lint, run tests                       |
| `Notification`     | Claude needs your attention | Desktop notifications                              |
| `Stop`             | Claude finished responding  | Run validators; auto-continue if work isn't done   |
| `SessionEnd`       | Session ends                | Cleanup                                            |
rest (`Setup`, `PermissionRequest`, `PostToolBatch`, `SubagentStart/Stop`, `PreCompact`, `FileChanged`, `CwdChanged`, etc.) you can learn as you need them.
### 03.Matcher
`matcher` filters _which occurrences_ of the event your handler responds to. For tool events, matcher filters on the tool name.

| Matcher                 | How it's interpreted                        |
| ----------------------- | ------------------------------------------- |
| `""`, `"*"`, or omitted | Match every occurrence                      |
| Letters/digits/`_`/`    | ` only                                      |
| Anything else           | JavaScript regex. `mcp__.*` = all MCP tools |
>for specific usage of Inside a single handler you can add `if` to filter on tool _arguments_, not just the name. It uses permission rule syntax:
>
>```jsonc
{
  "type": "command",
  "if": "Bash(git *)",         // only fires for git subcommands inside Bash
  "command": "./check-git.sh"
}
>```

### 04.How your script talks to Claude Code

**Input** arrives on stdin as JSON. For a `PreToolUse` hook on Bash, you'd get:

json

```json
{
  "session_id": "abc123",
  "cwd": "/Users/sarah/myproject",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "npm test" }
}
```

**Output** uses stdout, stderr, and exit codes:

| Exit code     | Meaning                                                                              |
| ------------- | ------------------------------------------------------------------------------------ |
| `0`           | Success. Claude Code parses stdout as JSON for fine control                          |
| `2`           | **Blocking error.** stderr is shown to Claude as feedback. The tool call is canceled |
| anything else | Non-blocking error. Action proceeds. stderr logged                                   |
## 05 — Where to put your personal hooks

Since you said "hooks only for me":

```
%USERPROFILE%\.claude\
├── settings.json          ← register hooks here
└── hooks\                 ← put your scripts here
    ├── format-if-js.js
    ├── audit-bash.ps1
    └── inject-git-state.ps1
```

And reference them with `${CLAUDE_PROJECT_DIR}` for project-rooted scripts, or absolute paths / `%USERPROFILE%` for user-level ones.
